# Общие правила разработки

Отвечай на русском языке. Комментарии к коду на русском. Используй русский язык.

---

## Универсальные принципы (любой язык)

- Один файл — одна ответственность. Большие файлы разбивать на модули.
- Каждая функция выполняет одну логическую задачу, длина до 40 строк.
- Следовать DRY, KISS, YAGNI — не дублировать, не усложнять, не писать лишнее.
- Избегать глубокой вложенности (не более 3 уровней).
- Предпочитать явные условия неявным.
- Использовать осмысленные имена — они должны отражать назначение.
- Константы вместо «магических чисел».
- Конфигурационные значения выносить в настройки/env.
- Разделять бизнес-логику и инфраструктурный код.
- Писать код, удобный для тестирования.
- Предпочитать ясность производительности, если разница незначительна.
- Оптимизировать только при реальной необходимости.

### Импорты
- Порядок: стандартные библиотеки → сторонние → локальные модули.
- Не использовать wildcard-импорты.
- Не оставлять неиспользуемые импорты.
- Предпочитать абсолютные импорты.

### Обработка ошибок
- Всегда обрабатывать возможные ошибки.
- Использовать исключения вместо кодов ошибок.
- Ловить только конкретные типы исключений.
- Не использовать пустые блоки catch/except.
- Сообщения об ошибках должны быть понятными.

### Логирование
- Использовать логирование вместо print/console.log в приложениях.
- Применять уровни: debug, info, warning, error.
- Не записывать чувствительные данные в логи.

### Комментарии и документация
- Комментарии объясняют причину решения, а не повторяют код.
- Писать docstrings/JSDoc для всех публичных функций и классов.
- Документация краткая и понятная, на русском языке.

## Python

### Стиль
- Строго придерживаться PEP8, документацию по PEP257.
- Длина строки до 100 символов.
- Переменные и функции — `snake_case`, классы — `PascalCase`, константы — `UPPER_CASE`.
- Булевые переменные начинать с `is_`, `has_`, `can_`, `should_`.
- Проверять код линтером (`ruff` или `flake8`), автоформатирование `black`.
- Сортировка импортов через `isort`.

### Типизация
- Всегда использовать аннотации типов (type hints).
- Современные типы: `list[str]`, `dict[str, int]` (не `List`, `Dict`).
- Для структурированных данных — `dataclass`, для сложных словарей — `TypedDict`.
- Не оставлять параметры без типов.

### Архитектура
- Использовать `pathlib` вместо строковых путей.
- Использовать context managers для работы с ресурсами.
- Избегать глобальных переменных и скрытых сетевых вызовов внутри функций.
- Использовать `dataclass` для простых моделей данных.
- Не изменять входные аргументы функций без необходимости.

### Асинхронное программирование (Python)
- Весь I/O-bound код писать через `async/await` — сеть, файлы, БД, внешние API.
- Никогда не вызывать `asyncio.run()` внутри уже запущенного event loop — использовать `await`.
- Синхронные блокирующие операции (gspread, Chroma, файловый I/O) выполнять через `loop.run_in_executor(None, fn)`.
- Узлы LangGraph, инструменты (`@tool`), обработчики FastAPI — всегда `async def`.
- Не смешивать синхронный и асинхронный код без явной причины и без executor.
- Инициализацию долгоживущих ресурсов (граф, клиенты) выполнять в `lifespan` FastAPI, не при первом запросе.
- Соединения, требующие `async with` (aiosqlite и др.), открывать на время запроса — не хранить как синглтон.

## Vue / TypeScript

### Компоненты
- Каждый компонент имеет одну ответственность.
- Файл компонента не превышает 150 строк — разбивать на подкомпоненты.
- Логику вкладок, секций, панелей выносить в отдельные компоненты.
- Использовать `<script setup lang="ts">` для всех компонентов.
- Компоненты и файлы именовать в `PascalCase`.
- Папки компонентов организовывать по функциональности (например: `common/`, `layout/`).
- Каждый компонент самодостаточен и переиспользуем.
- Избегать глубокой вложенности шаблона (не более 4 уровней).
- Не использовать inline-стили — только CSS-классы (Tailwind или аналог).
- Модальные окна всегда оборачивать в `<Teleport to="body">` — иначе `overflow`/`stacking context` родительских контейнеров ломают позиционирование и размеры.

### Логика
- Props и emits типизировать через TypeScript интерфейсы.
- Не писать бизнес-логику прямо в шаблоне — выносить в `computed` или функции.
- Composables называть с префикса `use`, использовать для переиспользуемой логики.

### Асинхронное программирование (TypeScript)
- Все запросы к API выполнять через `async/await`, не использовать `.then()/.catch()` цепочки.
- Обрабатывать ошибки через `try/catch` — не игнорировать отклонённые Promise.
- Не блокировать UI во время запроса — использовать флаги `isLoading`/`isStreaming`.
- SSE-стриминг читать через `ReadableStream` + `getReader()`, не через `EventSource` (не поддерживает POST и заголовки).
- Параллельные независимые запросы запускать через `Promise.all()`, не последовательно.

### Состояние (Pinia)
- Глобальное состояние хранить в Pinia stores (`src/stores/`).
- Каждый store отвечает за одну предметную область.
- Stores именовать с префикса `use` и суффикса `Store` (например `useSessionsStore`).
- Использовать composition API стиль (`defineStore('id', () => { ... })`).
- Локальное состояние компонента держать в `ref`/`reactive` внутри компонента — не выносить в store без необходимости.
- Не обращаться к store напрямую из шаблона — через composable или computed.

### Структура
- Типы и интерфейсы выносить в `src/types/`.
- Константы выносить в отдельный файл утилит (например `src/utils/constants.ts`).
- Переменные и функции — `camelCase`, классы/компоненты — `PascalCase`, константы — `UPPER_CASE`.
- Булевые переменные начинать с `is`, `has`, `can`, `should`.
